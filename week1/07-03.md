# be-DailyJob 

### **1. 오늘 완료한 것**

- MIME 타입 학습
- Content-Type 설정해서 response 하는 기능 구현

### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단: 스레드 풀에 작업을 전달하는 방식을 submit()에서 execute()로 변경
- 그렇게 판단한 이유: submit()은 작업 결과를 Future 객체로 반환받아 추적하거나 예외를 처리할 때 유용하다. 지금 내가 구현하는 WAS는 요청 처리를 별도의 스레드에 위임만 하면 되기 때문에 Future 객체를 별도로 만들지 않는 execute()가 더 적절하다고 생각했다.

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황 : HTTP 응답을 구현하면서 Content-Type 헤더를 어떤 기준으로 결정해야 할지 고민이 되었다.
- 대안 1 : 클라이언트의 Accept 헤더를 기준으로 Content-Type을 결정하는 방법을 생각했다. 하지만 클라이언트가 요청한 타입과 서버가 보유한 리소스의 실제 타입이 다를 수 있다는 점이 걸렸다. 또한 Accept 헤더에는 여러 MIME 타입이 포함될 수 있어, 서버가 지원 가능한 타입을 우선순위에 따라 확인하는 추가적인 처리가 필요하다고 생각했다.
- 대안 2 : 서버가 제공하는 파일의 실제 타입을 기준으로 Content-Type을 결정하는 방법을 생각했다. 이 경우 응답 데이터의 타입을 명확하게 표현할 수 있고, 서버가 실제로 제공하는 리소스의 정보와도 일치한다는 장점이 있었다.
- 결론: HTTP 응답 헤더에 대해 학습하면서 Accept와 Content-Type의 역할이 다르다는 점을 알게 되었다. 처음에는 Accept 헤더를 클라이언트가 원하는 타입을 전달하는 용도로만 생각해 서버도 해당 값에 맞춰 응답해야 한다고 생각했다. 하지만 Accept는 클라이언트가 처리 가능한 데이터 타입을 서버에 알려주는 요청 헤더이고, Content-Type은 서버가 실제로 반환하는 데이터의 타입을 나타내는 응답 헤더였다. 그래서 서버에 존재하는 파일의 확장자를 기준으로 MIME 타입을 결정하고, 이를 Content-Type으로 설정하는 방식이 적절하다고 판단했다.

### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지: MIME 타입을 저장하는 방식을 결정하기 전에 여러 자료구조를 후보로 두고 비교해봤다.
- 그 접근이 도움이 된 이유: 여러 선택지를 비교하는 과정에서 각 방법의 장단점을 이해할 수 있었고, 현재 요구사항에 가장 적합한 방식을 근거를 가지고 선택할 수 있었다.
- 다음에도 반복하고 싶은 점: 하나의 해결책이 떠오르더라도 바로 구현에 들어가기보다 다른 선택지는 없는지 먼저 검토하는 습관을 유지하고 싶다.

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

첫번째

- 아쉬웠던 점 : 처음 보는 패키지나 기술을 사용할 때 충분히 학습하지 않고 바로 구현부터 시작하는 경우가 있다.
- 이유 : 스쿼드 세션에서 다른 분들은 새로운 패키지를 접하면 해당 패키지가 어떤 역할을 하는지, 어떤 기능을 제공하는지 먼저 살펴보며 학습한다고 하셨다. 나는 기능 구현에 집중하는 경우가 많았는데, 새로운 기술이나 패키지를 만났을 때는 그 목적과 사용 방법을 먼저 이해하려는 노력이 필요하다고 느꼈다.
- 어떻게 개선할 것인지 : 처음 접하는 기술이나 라이브러리를 사용할 때는 구현에 들어가기 전에 공식 문서를 읽어보고, 해당 패키지가 어떤 역할을 하는지와 어떤 기능을 제공하는지 먼저 파악하는 습관을 들이고 싶다.

두번째

- 아쉬웠던 점 : 지금까지 구현하면서 애매한 부분이 생기면 AI나 주변 동료에게 의견을 묻는 경우가 많았는데, RFC 문서를 직접 확인해볼 생각은 하지 못했다.
- 어떻게 개선할 것인지 : 피드백 시간에 HTTP 규칙은 엄격하게 정의되어 있다는 말씀을 들었다. 그래서 앞으로는 구현하면서 애매한 부분이 생기면 RFC 문서를 먼저 살펴보는 습관을 가져야겠다.
