### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단: HttpRequest와 HttpResponse에 원래는 상태와 행동(파싱, 응답 생성 등)을 모두 포함시키는 구조였지만, 이를 각각 순수 데이터 객체로 분리하고 Parser / Builder를 도입하여 책임을 분리하는 구조로 변경했다.
- 그렇게 판단한 이유:
    - 현재 구조에서는 HttpRequest / HttpResponse가 데이터 모델 역할, 파싱/생성 로직 역할 두 가지 책임을 동시에 가지고 있어 변경 이유가 섞이는 문제가 있었다.
    - 예를 들어, HTTP 파싱 규칙이 바뀌거나 객체 구조가 바뀌면 동일 클래스가 함께 수정되어야 하는 구조였다.
    - 따라서 단일 책임 원칙(SRP)에 따라 데이터는 객체가 담당, 변환/생성 로직은 Parser / Builder가 담당으로 분리하는 것이 더 적절하다고 판단했다.
- 고려했지만 선택하지 않은 대안:
    - HttpRequest.from(InputStream) 형태로 기존처럼 팩토리 내부에 파싱 로직 유지
    - → 구현은 단순하지만 책임이 섞여 있어 테스트 및 확장성이 떨어짐
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - HttpRequest / HttpResponse가 순수 데이터 구조로 명확해짐
    - Parser / Builder 단위로 독립적인 테스트 작성이 가능해짐
    - HTTP 파싱과 도메인 모델 변경이 서로 영향을 주지 않게 되어 결합도가 감소함

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황 : `http://localhost:8080/` 로 접속했을 때 404 Not Found 대신, static 디렉토리 내부의 파일 및 디렉토리 이름이 브라우저에 그대로 출력되는 현상이 발생했다.
- 처음 추측한 원인: 요청 path가 /로 들어오면서 /static/ 디렉토리를 그대로 가리키게 되었고, 이 디렉토리 내부의 리소스들이 반환되고 있다고 추측했다. 디렉토리 여부를 구분하지 못하는 로직 문제라고 생각했다.
- 원인을 확인하기 위해 시도한 방법: /aa처럼 다른 경로로 요청을 보내봤을 때는 404 Not Found가 정상적으로 반환되는 것을 확인했다. 따라서 존재하는 디렉토리의 이름으로 요청했을 때 문제라고 판단했다.
- 실제 원인:  리소스를 읽는 과정에서 InputStream 기반으로 처리하고 있었는데, 이 방식은 파일과 디렉토리를 구분할 수 없다. 그 결과 디렉토리 경로가 들어와도 동일하게 읽히면서 디렉토리 목록이 그대로 응답으로 노출되었다.
- 해결 방법: File 객체 기반으로 로직을 변경하고, File.isDirectory() 를 통해 디렉토리 여부를 명확하게 검사하도록 수정했다. 이를 통해 디렉토리 접근 시에는 파일로 처리하지 않고 404 처리를 하도록 개선했다.

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

- 비효율적이었던 접근/놓치거나 부족했던 점: 디렉토리 문제를 해결하는 과정에서 File 객체를 사용해 처리한 부분
- 그렇게 된 원인: 디렉토리 여부를 판단하기 위해 File을 사용했는데, 동시에 배포 환경에서도 동작하도록 하기 위해 ResourceLoader.class.getResourceAsStream() 기반으로 리소스를 읽도록 설계해놓고, 실제 파일 처리 단계에서는 OS 파일 시스템 기반의 File 객체를 사용하면서 두 접근 방식이 섞여 일관성이 깨진 상태가 되었다.
- 다음에는 다르게 해볼 구체적인 행동: 리소스 접근 방식을 ClassPath 기반으로 유지할지, File 시스템 기반으로 가져갈지 기준을 명확히 정하고, 해당 기준에 맞춰 일관된 방식으로 처리하도록 다시 정리하고 적용해봐야겠다.
