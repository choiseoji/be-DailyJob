### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - HttpSession 저장소의 자료구조를 HashMap에서 ConcurrentHashMap으로 변경했다.
- 그렇게 판단한 이유
    - 세션 저장소는 여러 요청을 처리하는 스레드들이 동시에 접근할 수 있는 공유 자원이라고 판단했다.
    - HashMap을 사용할 경우 동시 접근 상황에서 데이터 일관성 문제가 발생할 수 있다고 생각했고, 이를 방지하기 위해 동시성 제어를 지원하는 ConcurrentHashMap을 선택했다.
- 고려했지만 선택하지 않은 대안
    - 처음에는 일반적인 HashMap을 사용하는 방안을 고려했다.
    - 단일 스레드 환경에서는 문제가 없지만, WAS는 여러 요청을 동시에 처리하기 때문에 세션 저장소로 사용하기에는 적합하지 않다고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - 현재는 User 객체 하나만 저장하고 있지만, 이후 세션에 여러 Attribute를 저장하거나 세션 데이터의 추가, 삭제 기능이 확장될 경우를 고려하면 동시 접근에 대한 기본적인 안전성을 확보할 수 있다.

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황
    - 기존에는 HttpResponseGenerator에서 상태 코드별로 응답을 생성하는 메서드를 만들어두었다.
        
        ```java
        public static HttpResponse redirect(String location) {
        
            return build(HttpStatus.FOUND, new byte[0], Map.of("Location", location));
        }
        ```
        
    - 로그인 기능을 구현하면서 로그인 성공 후 리다이렉트 응답과 함께 `Set-Cookie` 헤더를 추가해야 하는 요구사항이 생겼다.
    - 기존 구조에서는 이를 처리하기 위해 헤더를 추가로 받는 redirect() 메서드를 오버로딩해야 했다.
    - 하지만 이후에도 다양한 응답 헤더와 옵션이 추가될 경우, 상태 코드별 메서드마다 여러 오버로딩이 계속 증가하는 구조가 될 것이라고 판단했다.
- 해결 방법
    - HttpResponseGenerator를 제거하고, HttpResponse에서 기본적인 응답 생성 메서드를 제공하도록 개선했다.
    - 응답 생성 시 항상 필요한 상태 코드, 기본 헤더 또는 바디와 같은 공통적인 요소는 정적 메서드로 제공하고, Set-Cookie와 같이 상황에 따라 추가되는 헤더 값은 메서드 체이닝 방식으로 외부에서 조합할 수 있도록 변경했다.
    
    ```java
    return HttpResponse.redirect("/index.html")
                  .addHeader("Set-Cookie", "sid=" + session.getId() + "; Path=/");
    ```
    
- 고려했지만 선택하지 않은 대안
    - Builder 패턴을 도입하는 방법도 고려했다.
    - Builder를 사용하면 응답의 각 요소를 유연하게 조합할 수 있다는 장점이 있지만
    - 모든 응답을 생성할 때마다 Builder 객체를 생성하고 조립하는 코드를 쓰는건 호출부의 중복을 증가시킨다고 판단했다.

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

- 비효율적이었던 접근/놓치거나 부족했던 점
    - Handler에서 DB 접근(DB 계층)과 HttpSessionManager(WAS 계층) 접근이 함께 이루어지는 구조가 아쉬웠다.
- 그렇게 된 원인
    - 코드를 구현하는 과정에서 import 문을 통해 클래스와 패키지 간의 의존 관계를 확인하지 않았던 것이 원인이라고 생각한다.
- 다음에는 다르게 해볼 구체적인 행동
    - Handler 계층도 세분화하여 클래스를 분리하고, Service 계층을 도입하는 방향으로 개선해볼 예정이다.
    - 이를 통해 Handler 계층에서만 WAS 관련 의존성을 관리하고, Service 계층에서는 DB 계층만 호출하도록 의존 방향을 정리하는 방향을 고민해보려고 한다.

+) 팀원 분이 해당 부분에 대해 질문을 주셔서 클래스와 계층 간 의존 관계에 대해 다시 고민해볼 수 있었다. 놓치고 있던 부분을 발견할 수 있었고, 좋은 피드백을 주신 팀원 분께 감사하다.
