# be-DailyJob 

### **1. 오늘 완료한 것**

- 동적, 정적 요청 라우팅 기능 구현
- 전역적인 예외 처리기 구현

<br>

### **2. 가장 막혔던 문제와 해결 과정**

동적 요청과 정적 요청을 처리하는 라우팅 구조를 설계하면서 응답 생성 책임과 Handler 추상화 기준에 대해 고민했다.

**1️⃣ HttpResponse 생성 책임**

**문제 상황**

- 동적 요청과 정적 요청의 HttpResponse 생성 방식이 달랐다.
    - 동적 요청: Controller에서 비즈니스 로직 수행 후 HttpResponse 생성
    - 정적 요청: ResourceLoader에서 파일 데이터를 반환하고 Router에서 HttpResponse 생성
- 동적 요청과 정적 요청을 처리해서 응답을 반환받는데 대칭적이지 않다고 느꼈다.

<br>

**대안 1. Controller에서는 응답 데이터만 반환하고 별도 계층에서 HttpResponse 생성**

- 장점
    - 동적 요청과 정적 요청의 반환 타입 통일 가능 (모두 데이터 반환으로)
- 문제점
    - 동적 요청에 대한 응답은 HTTP 상태 코드, 헤더 등을 제어하는게 중요한데, 이런 정보를 중간 계층에서 처리하기 어려움

→ Controller에서 HTTP 상태 코드, 헤더를 제어하는게 좋다고 판단

<br>

**대안 2. 정적 요청 처리 계층 추가**

- ResourceLoader는 파일 조회 역할만 담당
- StaticResourceHandler 중간 계층을 추가하여 정적 요청 처리 책임 분리

```
StaticResourceHandler
        |
        ├── ResourceLoader (파일 조회)
        |
        └── HttpResponseGenerator (응답 생성)
```

- Router가 리소스 조회 방식과 응답 생성 방식까지 알 필요가 없어짐
- Router에서는 DynamicRequestHandler 또는 StaticResourceHandler만 호출하는 구조가 됨

<br>

### **3. 오늘 잘한 행동**

- 원하는 구조를 찾을 때까지 충분히 고민한 점.
    - 직무 과제 기간은 메인 프로젝트를 진행하기 전에 충분히 학습하고 고민하는 시간이라고 말씀해주셨기 때문에, 앞으로도 내가 납득할 수 있고 유지보수하기 좋은 구조를 찾을 때까지 충분히 고민하며 과제를 수행하고 싶다.
- 고민하고 있는 부분을 팀원들에게 공유한 점.
    - 다른 사람들의 의견을 들어보면서 내가 생각하지 못했던 관점과 새로운 방향을 얻을 수 있었다.

<br>
