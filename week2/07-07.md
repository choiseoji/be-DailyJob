### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - WAS와 Application의 책임을 분리하고, 동적 요청 처리는 app 패키지의 Handler가 담당하도록 설계했다.
    - 개발자가 HandlerConfig에 Handler 정보를 쓰면, WAS는 이를 주입받아 동작하는 방향으로 리팩토링했다.
- 그렇게 판단한 이유
    - WAS는 요청을 받아 적절한 Handler를 실행하는 역할에 집중해야 한다고 생각했다.
    - 만약 Handler 구현체가 WAS 내부에 위치하거나 WAS가 Handler 정보를 직접 관리한다면, 애플리케이션 개발자가 WAS의 내부 구조를 이해해야 한다.
    - 이는 WAS와 애플리케이션의 결합도를 높이고 역할의 경계를 모호하게 만든다고 판단했다.
- 고려했지만 선택하지 않은 대안
    - Handler 구현체를 WAS 패키지 내부에 두고, WAS가 직접 Handler 목록을 생성 및 관리하는 구조를 고려했다.
    - 새로운 Handler가 추가될 때마다 WAS 코드가 수정되어야 하고, WAS를 재사용 가능한 실행 환경으로 보기 어려워진다고 생각했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - 개발자가 새로운 Handler 추가 시 WAS 쪽의 코드를 수정할 필요가 없어졌다.
    - WAS는 Handler를 실행하기 위한 인프라 역할에만 집중하고, 애플리케이션은 비즈니스 로직 구현에만 집중할 수 있게 되었다.

### **3. 가장 막혔던 문제와 해결 과정**

- 상황
    - 현재 HttpRequest를 파싱할 때 BufferedReader를 사용하고 있다.
    - 하지만 HTTP Body는 항상 문자열이라는 보장이 없고, 이미지와 같은 바이너리 데이터가 전달될 수도 있다.
    - 따라서 Body는 문자열이 아닌 byte[] 형태로 보관하는 것이 적절하다고 판단했다.
- 고민한 부분
    - InputStream을 사용하면 Body를 byte[]로 읽을 수 있지만, 단순히 1바이트씩 읽는 방식은 비효율적일 수 있다.
    - 반면 BufferedReader는 내부 버퍼를 사용하기 때문에 문자열 데이터를 읽을 때 성능상 이점이 있다.
    - 그래서 요청 라인과 헤더는 BufferedReader로 읽고, Body만 InputStream으로 읽는 방법도 고려했다.
    - 하지만 BufferedReader가 데이터를 읽는 과정에서 Body의 일부가 내부 버퍼에 미리 저장될 수 있다.
    - 이런 상태에서 다시 InputStream으로 Body를 읽으면 데이터가 누락되거나 예상과 다르게 동작할 가능성이 있다고 판단했다.
- 해결 과정
    - 요청 전체를 InputStream 기반으로 처리하는 방향을 선택했다.
    - 요청 라인과 헤더는 바이트 데이터를 문자열로 변환하여 파싱하고, Body는 원본 byte[] 그대로 보관하도록 수정했다.
    - 이를 통해 문자열 데이터와 바이너리 데이터를 모두 동일한 방식으로 처리할 수 있게 되었다.
- 느낀점
    - Body를 byte[]로 저장하는 것까지는 해결했지만, 이를 언제 어디서 사람이 읽을 수 있는 형태로 변환할지에 대한 고민은 아직 남아 있다.
    - 현재는 Handler가 직접 적절한 파서를 선택하여 Body를 해석하도록 구현했다.
    - 하지만 Handler가 파서의 종류를 알지 못한 채 필요한 데이터만 얻을 수 있도록 추상화할 방법이 있을 것 같아, 앞으로 더 고민해 볼 예정이다.

### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지
    - 어제 팀원분의 코드를 보면서 마음에 드는 구조를 발견했다.
    - 해당 구조를 차용했을 때의 장단점을 먼저 고민해보고, 팀원과 함께 설계 의도와 적용 방향에 대해 토의하는 과정을 거쳤다.
- 그 접근이 도움이 된 이유
    - 토의를 진행하면서 내가 작성한 코드를 더 나은 구조로 리팩토링할 수 있었다.
    - 또한 팀원분이 지속적으로 설계 이유를 질문해 주셔서 내가 내린 판단을 다시 검토해 볼 수 있었고, 그 과정에서 구조에 대한 이해도도 높아졌다.
    - 설계 의도를 공유하며 의견을 나눈 점이 의미 있었다.
- 다음에도 반복하고 싶은 점
    - 코드 리뷰를 할 때 배울 점을 찾는 관점으로 꼼꼼하게 살펴보려고 한다.
    - 또한 설계 의도가 궁금한 부분이 있다면 적극적으로 질문하고, 내 생각도 함께 공유하며 토의를 이어가는 습관을 유지하고 싶다.

</br>
