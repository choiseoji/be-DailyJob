### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - 하나의 `PageHandler`가 여러 페이지의 렌더링을 담당하던 구조를 도메인별 `PageHandler`로 분리하고, 공통 렌더링 로직은 `BasePageHandler`로 추출했다.
- 그렇게 판단한 이유
    - 기존 구조는 하나의 핸들러가 여러 페이지의 렌더링을 담당하고 있어 책임이 과도하게 집중되어 있었다.
    - 페이지마다 필요한 데이터와 비즈니스 로직은 달랐지만, 실제 템플릿을 렌더링하는 과정은 대부분 동일했다.
    - 공통 로직과 페이지별 책임을 분리하면 코드의 역할이 더 명확해질 것이라고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - 페이지별 책임이 명확해져 SRP(단일 책임 원칙)를 더 잘 지킬 수 있게 되었다.
    - 새로운 페이지가 추가되더라도 기존 핸들러를 수정하기보다 새로운 핸들러를 추가하면 되어 확장성이 좋아졌다.
    - 각 핸들러를 독립적으로 테스트할 수 있어 테스트 범위를 더 명확하게 가져갈 수 있게 되었다.

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황 또는 증상
    - 한국어 파일명을 가진 이미지는 서버에 정상 저장되었지만 브라우저에서 이미지를 요청하면 404 Not Found가 발생했다.
- 처음 추측한 원인
    - 파일 저장 과정에서 인코딩이 깨졌거나 파일명이 잘못 저장된 것이라고 생각했다.
- 원인을 확인하기 위해 시도한 방법
    - 실제 저장된 파일명을 확인했다.
    - 브라우저 개발자 도구(Network 탭)에서 이미지 요청 URL을 확인했다.
    - 서버 로그를 출력하여 요청으로 들어오는 경로와 실제 저장된 파일명을 비교했다.
- 실제 원인
    - 브라우저는 한국어가 포함된 URL을 퍼센트 인코딩(`%EC%8A%A4...`) 형태로 변환하여 요청한다.
    - 서버는 인코딩된 문자열 그대로 파일 경로를 찾고 있었기 때문에 실제 저장된 파일명과 일치하지 않아 404가 발생했다.
- 해결 방법
    - 업로드 이미지를 조회하는 로직에서 요청 경로를 `URLDecoder`로 디코딩한 뒤 파일을 조회하도록 수정했다.
    - 이를 통해 브라우저 요청 경로와 실제 파일명이 일치하도록 만들었다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것
    - 브라우저가 요청 시 URL 인코딩을 수행하는지 확인한다.
    - 서버가 요청 경로를 어떤 형태로 받고 있는지 로그로 출력해본다.
    - 저장된 데이터와 요청 데이터가 동일한 인코딩 규칙을 사용하는지 먼저 점검한다.
    - 문자열 자체보다 인코딩/디코딩 과정에서 값이 달라지는 것은 아닌지 확인한다.

### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지
    - 이미지 업로드 기능을 구현할 때 바로 자료를 찾아보거나 예제 코드를 참고하기보다, 먼저 직접 multipart/form-data 요청을 보내고 로그를 확인하며 요청 구조를 분석했다.
    - boundary가 어떻게 전달되는지, 각 파트가 어떤 형식으로 구성되는지 직접 확인한 뒤 파싱 로직을 설계했다.
- 그 접근이 도움이 된 이유
    - 동작 원리를 먼저 이해한 상태에서 구현을 시작할 수 있어 파싱 로직을 설계하기가 수월했다.
    - 단순히 예제를 따라 작성하는 것이 아니라 HTTP 요청이 실제로 어떤 형태로 전달되는지 이해할 수 있었고, 구현 과정에서 발생하는 문제도 더 빠르게 파악할 수 있었다.
- 다음에도 반복하고 싶은 점
    - 새로운 기능을 구현할 때는 관련 자료를 바로 찾아보기 전에 실제 데이터를 확인하고 동작 원리를 먼저 파악해보는 습관을 유지하고 싶다.
