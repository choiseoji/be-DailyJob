### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - 게시글 상세 조회 시 현재 게시글, 이전 게시글, 다음 게시글을 조회하기 위해 여러 번의 쿼리가 발생하던 구조를 하나의 조회 쿼리로 통합했다.
    - 기존에는 게시글 조회에서 `findPrev()`, `findNext()`, `findById()`를 각각 호출하면서 게시글 하나를 조회하는데 총 3번의 쿼리가 발생했다.
    - 이를 `UNION ALL`을 활용해 현재 게시글과 이전/다음 게시글 정보를 한 번에 조회하도록 변경했다.
- 그렇게 판단한 이유
    - 게시글 상세 조회는 사용자가 자주 접근하는 기능이므로, 하나의 요청마다 여러 번의 DB 접근이 발생하는 구조는 개선할 필요가 있다고 판단했다.
    - 필요한 데이터를 하나의 쿼리에서 조회하도록 변경하여 DB 접근 횟수를 줄이고, 네트워크 비용과 DB 부하를 줄일 수 있다고 판단했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - 게시글 상세 조회 과정에서 발생하는 DB 접근 횟수를 줄여 조회 성능을 개선할 수 있었다.
    - 기존에는 여러 Repository 호출을 통해 데이터를 조합했지만, Repository에서 상세 조회에 필요한 데이터를 한 번에 조회하도록 변경하면서 조회 로직의 책임을 명확히 했다.
    - 다만 단순한 CRUD 쿼리보다 SQL 복잡도가 증가했기 때문에, 조회 결과 매핑과 이전/다음 게시글 조회 조건에 대한 테스트가 필요해졌다.

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황
    - Path Variable을 어디에서 추출하고 저장해야 하는지 결정하는 과정에서 고민이 있었다.
    - 처음에는 `HttpRequestParser`에서 Path Variable을 추출하려 했지만, Parser가 URL 패턴(`/posts/{postId}`) 정보를 알고 있어야 하기 때문에 HTTP 요청 파싱이라는 본래 책임과 맞지 않다고 판단했다.
    - 이후 URL 패턴과 실제 요청 Path를 비교하는 역할은 `HandlerMapping`의 책임이라고 판단하여, `HandlerMapping.find()` 과정에서 Path Variable을 추출하도록 변경했다.
    - 이에 따라 기존에 `Handler`만 반환하던 `HandlerMapping.find()`가 `HandlerResult(Handler, Path Variables)`를 반환하도록 수정했다.
- 해결 과정에서 발생한 문제
    - (기존 구조: `RequestHandler → RequestRouter → HandlerMapping`)
    - 추출한 Path Variable을 Handler에서 사용할 수 있도록 전달하는 방법에 대한 고민이 생겼다.
    - Path Variable을 `HttpRequest`에 저장해서 Handler가 사용할 수 있도록 하는 방식을 선택했지만, `HttpRequest`는 `Record` 타입으로 설계되어 있어 기존 객체를 직접 변경할 수 없었다.
    - 따라서 Path Variable이 추가된 새로운 `HttpRequest` 객체를 생성해야 했고, 이는 요청 객체를 생성하고 흐름을 관리하는 `RequestHandler`에서 처리하는 것이 적절하다고 판단했다.
    - 하지만 기존 구조에서는 `RequestRouter`가 `HandlerMapping`의 결과(`HandlerResult`)를 전달하는 역할만 수행하게 되었고, `RequestHandler`에서 다시 요청 객체를 조립해야 하는 구조가 되었다.
    - 결국 `RequestRouter`가 별도의 책임 없이 중간 전달 계층 역할만 하게 되었기 때문에, 해당 계층을 제거하고 `RequestHandler`에서 직접 `HandlerMapping`을 호출하는 구조로 변경했다.

### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지
    - 구현한 기능이 정상적으로 동작하는지 다양한 입력 조건으로 반복해서 테스트했다.
    - 그 과정에서 게시글에 개행이 정상적으로 출력되지 않는 문제를 발견하고 수정할 수 있었다.
- 그 접근이 도움이 된 이유
    - 기능 구현 자체에만 집중했다면 일반적인 한 줄짜리 데이터만 테스트하고 넘어가 해당 문제를 발견하지 못했을 것이다.
- 다음에도 반복하고 싶은 점
    - 기능 구현 후 정상 케이스뿐만 아니라 다양한 입력 형태와 예외 상황을 직접 테스트하는 습관을 유지하고 싶다.
    - 특히 문자열, 파일, 요청 데이터처럼 사용자의 입력이 들어오는 부분은 다양한 형태의 데이터를 넣어보며 검증하는 습관을 가져가고 싶다.

</br>

- 비효율적이었던 접근/놓치거나 부족했던 점

    - 이후 팀원이 Ajax가 무엇인지 질문했을 때, 들어본 개념이었지만 Ajax의 의미와 동작 방식을 명확하게 설명하지 못했다.
- 그렇게 된 원인
    - 백엔드 서버 구현과 구조 개선에 집중하다 보니 클라이언트 측 코드 흐름에 대한 이해를 깊게 하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동
    - 특히 `fetch`, Ajax와 같은 클라이언트 요청 방식이 HTTP 요청/응답 과정에서 어떻게 동작하는지 학습하고, 백엔드와 연결되는 전체 흐름을 이해하도록 노력하겠다.
