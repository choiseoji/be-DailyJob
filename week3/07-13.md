### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - 각 Repository마다 반복되고 있던 DB Connection 생성, PreparedStatement 생성, 예외 처리 등의 공통 로직을 `Repository` 최상위 클래스로 분리했다.
    - 이후 `PostRepository`, `UserRepository` 등은 `Repository`를 상속받아 공통 DB 실행 로직을 재사용하고, 각 도메인에 필요한 SQL 작성과 결과 매핑에 집중하도록 변경했다.
- 그렇게 판단한 이유
    - 기존에는 Repository마다 동일한 JDBC 실행 코드가 반복되고 있어 새로운 Repository가 추가될 때마다 같은 로직을 작성해야 했다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - Repository마다 반복되던 JDBC 실행 코드를 제거하여 중복을 줄이고 코드 가독성을 개선했다.
    - DB 연결 및 예외 처리 로직을 한 곳에서 관리할 수 있어 Repository 구현체들은 SQL과 도메인 데이터 변환에 집중할 수 있게 되었다.
    - 공통 로직이 변경될 경우 여러 Repository를 수정하지 않고 상위 클래스만 수정할 수 있도록 변경 범위를 줄였다

</br>

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황
    - Path Variable을 처리하는 과정에서 추출한 값을 Handler에서 어떻게 사용하도록 전달할지 고민이 있었다.
    - 현재는 각 Handler에서 직접 Path Variable을 추출하여 사용하는 구조인데, 동일한 로직이 반복될 가능성이 있어 공통화할 수 있는 방법을 고민하고 있다.
    - 라우팅 과정에서 Path Variable을 관리할지, 별도의 공통 처리 계층을 둘지에 대한 책임 분리가 아직 명확하지 않아 추가적인 개선이 필요하다고 판단했다.

</br>

- 다음에도 반복하고 싶은 점
    - 항상 익숙한 방식만 선택하기보다, 다양한 접근 방법을 고민하고 직접 경험해보는 습관을 유지하고 싶다.

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

- 비효율적이었던 접근/놓치거나 부족했던 점
    - 테스트 코드를 작성하는 과정에서 Service가 Repository를 의존하고 있는 구조에서 Repository를 어떻게 주입해야 하는지 고민하는 시간이 길어졌다.
    - Mock 라이브러리를 사용하면 Repository를 대체할 수 있지만, 현재 환경에서는 추가 의존성을 사용하지 않는 방향으로 진행하고 있었기 때문에 다른 방법을 고민했다.
    - 또한 테스트 환경에서 사용하는 H2 데이터베이스 설정을 정확히 이해하지 못해 운영 환경과 테스트 환경의 데이터베이스를 어떻게 분리해야 하는지 고민했다.
    - `DB_CLOSE_DELAY=-1` 옵션을 JVM이 종료된 이후에도 데이터가 유지되는 설정으로 착각했지만, 실제로는 모든 Connection이 닫혀도 JVM이 살아있는 동안만 메모리 DB를 유지하는 설정이었다.
    - 이를 이해한 후 테스트 코드에서는 별도의 Mock Repository를 만들 필요 없이 `new UserRepository()`로 실제 Repository 구현체를 주입하는 방식으로 해결했다.
- 그렇게 된 원인
    - 테스트 코드 작성 시 Service의 의존성을 어떻게 관리할지에 대한 경험이 부족했고, Mock 객체와 실제 Repository를 활용하는 테스트 방식에 대한 이해가 부족했다.
    - 또한 H2 In-Memory Database의 생명주기와 설정 옵션에 대한 이해가 부족해 테스트 환경 구성에 대한 고민이 길어졌다.
- 다음에는 다르게 해볼 구체적인 행동
    - 테스트 코드를 작성하기 전에 테스트 대상 객체의 의존 관계를 먼저 파악하고, 실제 객체를 사용할지 Mock 객체를 사용할지 판단하는 기준을 정리해야겠다.
    - 사용하는 데이터베이스 설정 옵션이 테스트 환경에 어떤 영향을 주는지 먼저 확인하고, 테스트 환경과 운영 환경을 분리하는 방법에 대해 더 학습하겠다.
