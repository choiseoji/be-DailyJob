### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - 정적 리소스 디렉터리 내부의 원본 파일을 외부에서 직접 조회하지 못하도록 접근을 제한했다.
- 그렇게 판단한 이유
    - 현재 서버 사이드 렌더링 방식을 사용하고 있기 때문에 템플릿 파일(`index.html`)을 직접 조회할 수 있다면 Mustache 문법(`{{ }}`)과 같은 템플릿 코드가 그대로 노출될 수 있다.
    - 또한 URL을 조작하여 정적 리소스 경로를 탐색할 경우 서버의 파일 구조나 내부 구현 방식이 외부에 노출될 가능성이 있다고 판단했다.
    - 사용자가 접근해야 하는 것은 렌더링된 결과 페이지이지 원본 템플릿 파일이 아니므로, 필요한 리소스만 제공하고 나머지는 차단하는 것이 더 안전하다고 생각했다.
- 고려했지만 선택하지 않은 대안
    - 모든 정적 파일 요청을 그대로 허용하는 방법도 고려했다.
    - 하지만 템플릿 파일이나 의도하지 않은 리소스가 노출될 수 있고, 서버 내부 구조에 대한 정보가 외부에 드러날 가능성이 있어 선택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - 외부에 노출되는 리소스 범위가 명확해져 보안 측면에서 더 안전한 구조가 되었다.
    - 사용자가 접근 가능한 리소스와 서버 내부 구현 파일의 경계가 분명해졌다.

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황 또는 증상
    - 존재하는 URL에 지원하지 않는 HTTP Method로 요청을 보냈을 때 405 Method Not Allowed를 반환해야 했지만, 현재 `HandlerMapping` 구조에서는 이를 구분하기 어려웠다.
- 처음 추측한 원인
    - 405 처리를 위한 분기 로직이 부족한 것이 문제라고 생각했다.
    - `Map<HandlerKey, Handler>` 구조를 유지한 채 예외 처리만 추가하면 해결할 수 있을 것이라고 예상했다.
- 원인을 확인하기 위해 시도한 방법
    - 현재 `HandlerMapping`의 조회 흐름을 분석하며, 요청 경로는 존재하지만 Method만 다른 경우를 어떻게 판별할 수 있는지 고민했다.
    - 기존 자료구조로는 결국 모든 매핑 정보를 순회해야 한다는 점을 확인했다.
- 실제 원인
    - 기존 구조는 `HandlerKey`가 `{HttpMethod, Path}` 조합으로 이루어져 있어 메서드와 경로가 모두 일치하는 경우는 빠르게 조회할 수 있었다.
    - 하지만 "경로는 존재하지만 Method가 다른 경우"처럼 일부 조건만 만족하는 상황을 확인하기 어려웠고, 결국 모든 매핑 정보를 순회해야 했다.
- 해결 방법
    - 매핑 구조를 `Map<String, Map<HttpMethod, Handler>>` 형태로 변경했다.
    - Path를 기준으로 먼저 조회한 뒤, 해당 Path에 등록된 Method를 확인하도록 조회 방식을 수정했다.
    - 이를 통해 경로가 존재하지 않는 경우는 404, 경로는 존재하지만 Method가 허용되지 않는 경우는 405를 명확하게 구분할 수 있게 되었다.
    - 또한 요청 처리 흐름이 URL 중심으로 정리되어 이후 라우팅 기능을 확장하기도 쉬워졌다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것
    - 특정 기능을 구현하기 위해 반복적인 순회나 복잡한 조건 분기가 필요하다면, 먼저 자료구조가 요구사항을 제대로 표현하고 있는지 확인한다.
    - 기능 추가 과정에서 현재 조회 방식이 요구사항과 맞지 않는다면, 로직을 덧붙이기보다 자료구조 자체를 변경하는 것이 더 적절한 해결책인지 검토한다.

### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지
    - 버그 데이가 진행되기 전에 직무 교육 조원들과 함께 미리 테스트를 진행하며 버그를 점검했다.
    - 발견된 버그는 미루지 않고 바로 수정하여 문제를 해결했다.
- 그 접근이 도움이 된 이유
    - 실제 버그 데이 전에 주요 문제들을 미리 발견하고 수정할 수 있었다.
    - 버그가 누적되지 않아 이후 원인 파악과 수정 비용이 줄어들었고, 최종적으로 버그 데이에서 새로운 버그가 발견되지 않았다.
- 다음에도 반복하고 싶은 점
    - 발견한 문제를 나중으로 미루지 않고 즉시 수정하는 방식이 품질 관리에 효과적이라는 것을 느꼈다.
    - 기능 구현뿐 아니라 검증 과정도 개발의 중요한 일부라고 생각하고 꾸준히 실천하고 싶다.

</br>

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

- 비효율적이었던 접근/놓치거나 부족했던 점
    - Javadoc을 작성해보려고 했지만, 사용 경험이 없다 보니 작성 방법과 실제로 어떻게 활용되는지 알아보는 데 시간을 많이 사용했고 결국 문서 작성을 마무리하지 못했다.
- 그렇게 된 원인
    - 지금까지 개인 프로젝트나 과제를 진행하면서 Javadoc을 적극적으로 사용해본 경험이 없었다.
    - 그래서 어떤 수준으로 작성해야 하는지, 어떤 정보를 담아야 하는지에 대한 기준이 부족했다.
- 다음에는 다르게 해볼 구체적인 행동
    - Javadoc의 기본 작성 방법과 활용 방식을 먼저 학습한 뒤 주요 클래스와 메서드부터 단계적으로 적용해볼 예정이다.
    - 앞으로는 기능 구현이 끝난 후 관련 문서를 함께 작성하는 습관을 들이고 싶다.
    - 특히 다른 개발자가 코드를 이해하는 데 도움이 되는 문서를 작성하는 연습을 꾸준히 해볼 계획이다.
