# be-DailyJob 

### **1. 오늘 완료한 것**

- 오늘 실제로 끝낸 기능/수정/학습
    - 기능 명세서 작성을 마무리했다.
    - 와이어프레임 초안을 검토하고 팀원들과 함께 피드백을 작성했다.
- (오늘의 목표 중에서) 아직 끝내지 못한 부분
    - 서비스의 프론트엔드 룩앤필(디자인 방향성) 선정

</br>

### **2. 오늘 가장 의미있었던 기술적 판단**

- 내가 내린 판단
    - 프론트엔드 폴더 구조를 Route Based Architecture로 결정했다.
- 그렇게 판단한 이유
    - 오늘 프론트엔드 특강에서 Route Based 구조의 장점에 대해 들을 수 있었다.
    - 이 구조는 일부 코드 중복이 발생할 수 있지만, 특정 화면을 수정했을 때 변경 범위가 해당 라우트 내부로 제한되는 경우가 많아 사이드 이펙트를 파악하기 쉽다는 장점이 있다.
    - 그동안은 코드 중복을 줄이는 것에 집중했는데, 유지보수성과 변경 영향도 파악 역시 중요한 기준이라는 점을 새롭게 배울 수 있었다.
- 고려했지만 선택하지 않은 대안
    - Feature Based Architecture
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향
    - 변경 범위를 추적하기 쉬워지고, 특정 기능 수정 시 예상치 못한 영향이 발생했는지 확인하기 수월해질 것으로 기대된다.

</br>

### **3. 가장 막혔던 문제와 해결 과정**

- 문제 상황 또는 증상
    - 경매방 메인 화면의 UI 구성 방향을 정하는 과정에서 의견 차이가 발생했다.
- 처음 추측한 원인
    - 팀원마다 머릿속에 그리고 있던 화면 구성이 달랐다.
    - 어떤 팀원은 모든 상품의 이벤트 로그를 한 화면에서 확인하는 구조를 생각했고, 다른 팀원은 상품별로 영역을 나누어 각각의 이벤트 로그를 보여주는 구조를 생각하고 있었다.
- 원인을 확인하기 위해 시도한 방법
    - 와이어프레임을 직접 수정하며 각자가 생각하는 화면을 시각적으로 표현했다.
- 해결 방법
    - 피그마 화면을 기준으로 서로의 생각을 맞춰가며 논의했고, 최종적으로 동일한 화면 구조를 공유할 수 있었다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것
    - 말로만 설명하기보다 간단한 스케치나 와이어프레임을 먼저 공유해 서로 같은 화면을 상상하고 있는지 확인하겠다.

</br>

### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지
    - 프론트엔드 특강에서 배운 내용을 팀원들과 바로 프로젝트에 적용해보았다.
    - Orval을 처음 사용해 보며 OpenAPI 문서를 기반으로 프론트엔드 코드를 자동 생성하는 과정을 경험했다.
- 그 접근이 도움이 된 이유
    - 평소에는 새로운 내용을 배우면 나중에 따로 정리하거나 적용하려고 미루는 경우가 많았다.
    - 이번에는 배운 내용을 즉시 실습하고 프로젝트에 적용하면서 개념을 더 빠르게 이해하고 익힐 수 있었다.
- 다음에도 반복하고 싶은 점
    - 새로운 기술이나 개념을 학습하면 가능한 한 바로 프로젝트에 적용해보는 습관을 유지하겠다.

</br>

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

- 비효율적이었던 접근/놓치거나 부족했던 점
    - 기획 관련 논의가 나왔을 때 더 좋은 방향을 제안하고 싶었지만, 아이디어가 부족해 명확한 대안을 제시하지 못했다.
- 그렇게 된 원인
    - 경매 도메인에 대한 이해도가 아직 충분하지 않아 사용자 관점이나 서비스 운영 관점에서 다양한 아이디어를 떠올리기 어려웠다.
- 다음에는 다르게 해볼 구체적인 행동
    - 기획 경험이 있는 주변 사람들과 이야기하며 문제를 바라보는 관점과 아이디어 도출 방식을 배워봐야겠다.
    - 또한 실제 경매 플랫폼과 관련 서비스를 직접 사용해보면서 사용자 경험과 운영 방식을 분석하고, 도메인에 대한 이해도를 높여야겠다.
