### **4. 오늘 잘한 행동(개발과 관련된 것)**

- 어떤 행동이었는지
    - 오늘 회의는 비교적 빠르고 효율적으로 진행되었다.
    - 모두가 하나의 논점을 중심으로 논의를 이어갔고, 중간에 다른 주제로 벗어나기보다 우선 핵심 의사결정을 내리는 데 집중하려고 노력했다.
- 그 접근이 도움이 된 이유
    - 회의를 하다 보면 하나의 주제에서 파생되는 여러 문제들이 함께 등장하면서 논의 범위가 넓어지고, 결과적으로 결정이 늦어지는 경우가 많다.
    - 오늘은 우선 핵심 문제에 대한 방향을 정한 뒤, 파생된 이슈들은 별도로 기록하는 방식으로 진행하여 의사결정 속도를 높일 수 있었다.
- 다음에도 반복하고 싶은 점
    - 앞으로도 회의에서는 먼저 핵심 안건에 대한 결론을 도출하고, 논의 과정에서 나온 파생 이슈들은 따로 정리해 두었다가 이후에 순차적으로 논의하는 방식을 유지하고 싶다.

</br>

### **5. 오늘 아쉬웠던 행동(개발과 관련된 것)**

- 비효율적이었던 접근/놓치거나 부족했던 점
    - 오늘 AWS 수업을 들으며 그동안 AWS 자원을 정확한 목적과 동작 원리를 이해하지 못한 채 사용해왔다는 것을 깨달았다.
    - 팀원분께서 NAT의 동작 원리와 네트워크 구성에 대해 설명해주셨는데, RDS가 인터넷 게이트웨이(IGW)를 통해 통신하는 경우에는 NAT가 필요하지 않으며, EC2를 Private Subnet에 배치할 경우에는 Public Subnet에 배스천 호스트(Bastion Host)나 NAT 인스턴스를 두는 방식으로 네트워크를 구성할 수 있다고 말씀해주셨다.
    - 이러한 설명을 들으며 그동안 AWS 서비스를 단순히 사용법 위주로만 익혔고, 각 자원이 왜 필요한지에 대한 이해는 부족했다는 점을 느꼈다.
- 그렇게 된 원인
    - AWS를 학습할 때 서비스 사용 방법과 설정 과정에만 집중했고, 각 자원의 역할과 네트워크 구조를 깊게 이해하려는 노력이 부족했다.
    - 기능 구현에만 집중하다 보니 인프라 구성의 이유와 동작 원리를 충분히 고민하지 못했다.
- 다음에는 다르게 해볼 구체적인 행동
    - 새로운 AWS 자원을 사용할 때는 단순히 설정 방법만 익히지 않고, 해당 자원이 어떤 문제를 해결하기 위해 존재하는지와 내부 동작 원리를 함께 학습하겠다.
    - VPC, Subnet, IGW, NAT Gateway/Instance 등 네트워크 관련 개념을 정리하며 전체 아키텍처 관점에서 이해하는 시간을 가져보겠다.
    - 자원 하나를 사용하더라도 "왜 필요한가?"를 먼저 고민하고 목적에 맞게 활용하는 습관을 들이겠다.
